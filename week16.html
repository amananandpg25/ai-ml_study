🧠 Theory

1. When to Fine-Tune vs. Prompt vs. RAG

ApproachWhen to UseCostLatency Impact
PromptingGeneral tasks, standard domains, quick iterationVery lowNone
RAGDynamic knowledge, large private corpora, facts that changeLow (storage)+retrieval time
Fine-TuningSpecific style/persona, domain jargon, highly structured output, lower inference cost at scaleHigh (one-time training)None (model is smaller)
💡 Decision Rule: Try prompting first. If that fails, try RAG. Only fine-tune when: (1) the task requires knowledge that can't be injected via context, (2) you need consistent output style across millions of calls, or (3) you want to distill a large expensive model into a small fast one.

2. LoRA — Low-Rank Adaptation

Full fine-tuning updates all model weights — for a 7B parameter model that's 28GB of gradient storage. LoRA observes that weight updates during fine-tuning have low intrinsic rank, so instead of updating the full weight matrix W, we add a small low-rank update:

Standard weight update (full fine-tuning): $$W' = W + \Delta W \quad \text{where } \Delta W \in \mathbb{R}^{d \times k}$$ LoRA approximation — decompose ΔW into two small matrices: $$\Delta W = BA \quad \text{where } B \in \mathbb{R}^{d \times r},\ A \in \mathbb{R}^{r \times k},\ r \ll \min(d, k)$$ At rank r=8, a 4096×4096 matrix: full ΔW = 16.7M params. LoRA ΔW = 2×(4096×8) = 65K params — a 256× reduction!
LoRA ko samjho jaise IAS exam ki preparation: poora 10 saal ka syllabus dobara padhna (full fine-tuning) vs sirf current affairs aur recent question patterns padhna (LoRA) — kyunki model ka base knowledge already solid hai, sirf ek specific "layer" add karni hai.
PYTHON — APPLYING LORA WITH PEFT
# pip install transformers peft accelerate datasets bitsandbytes
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model, TaskType
import torch

# Load base model (use a small one for demo)
model_name = "microsoft/phi-2"   # 2.7B params, great for experimentation
tokenizer  = AutoTokenizer.from_pretrained(model_name)
model      = AutoModelForCausalLM.from_pretrained(
    model_name, torch_dtype=torch.float16, device_map="auto"
)

# Define LoRA configuration
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,  # language model fine-tuning
    r=8,                            # rank — lower r = fewer params, faster but less expressive
    lora_alpha=32,                 # scaling factor: α/r = 4 (effective learning rate scale)
    target_modules=["q_proj", "v_proj"],  # which layers to apply LoRA to
    lora_dropout=0.05,             # regularisation
    bias="none"                     # don't add bias parameters
)

# Wrap the model with LoRA — only LoRA params will be trained
peft_model = get_peft_model(model, lora_config)
peft_model.print_trainable_parameters()
# trainable params: 2,097,152 || all params: 2,779,683,840 || trainable%: 0.08%
# Only 0.08% of parameters are trained!

# Inspect which modules have LoRA adapters
for name, module in peft_model.named_modules():
    if "lora" in name:
        print(name, "→", module.__class__.__name__)

3. QLoRA — 4-bit Quantisation + LoRA

QLoRA (Dettmers et al., 2023) lets you fine-tune a 65B model on a single 48GB GPU — or a 7B model on a free Colab T4 (15GB). The trick: load the frozen base model in 4-bit NormalFloat (NF4) quantisation, add 16-bit LoRA adapters on top. Gradients flow through LoRA (fp16) but base model stays 4-bit.

TechniqueMemory for 7B modelQuality vs Full FT
Full fine-tuning (fp16)~112 GBBaseline
LoRA (fp16 base)~14 GB~97% of full FT
QLoRA (4-bit base + fp16 LoRA)~6 GB~95% of full FT
PYTHON — QLORA SETUP WITH BITSANDBYTES
from transformers import BitsAndBytesConfig
from peft import prepare_model_for_kbit_training

# 4-bit quantisation config (NF4 = NormalFloat4, best quality)
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",          # NormalFloat4 — optimal for normally-distributed weights
    bnb_4bit_compute_dtype=torch.bfloat16, # compute in bf16 even though stored in 4-bit
    bnb_4bit_use_double_quant=True,       # quantise the quantisation constants too (saves ~0.4 bits/param)
)

# Load in 4-bit — only ~6GB for a 7B model!
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantization_config=bnb_config,
    device_map="auto",
)

# Prepare for k-bit training: cast layer norms to fp32, enable gradient checkpointing
model = prepare_model_for_kbit_training(model)

# Apply LoRA on top of the quantised model
lora_config = LoraConfig(
    r=16,
    lora_alpha=64,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],  # all attention projections
    lora_dropout=0.1,
    task_type=TaskType.CAUSAL_LM,
)
peft_model = get_peft_model(model, lora_config)
peft_model.print_trainable_parameters()
# trainable params: 33,554,432 || all params: 3,540,389,888 || trainable%: 0.95%

4. Dataset Preparation — Alpaca & ShareGPT Formats

Supervised Fine-Tuning (SFT) needs instruction-response pairs. The two dominant formats are Alpaca (simple) and ShareGPT (multi-turn conversation). Both ultimately get formatted into a single string that the model learns to predict.

PYTHON — DATASET FORMATTING FOR SFT
from datasets import Dataset

# --- Alpaca format ---
alpaca_examples = [
    {
        "instruction": "Explain gradient descent in Hindi in 2 sentences.",
        "input": "",   # optional extra context
        "output":     "Gradient descent ek optimization algorithm hai jo loss ko minimise karta hai. Ye har step mein gradient ke ulte direction mein weights ko update karta hai."
    }
]

def alpaca_to_prompt(sample: dict) -> str:
    """Format one Alpaca sample into a training string."""
    inp = (f"\n\n### Input:\n{sample["'input'"]}" if sample["\"input\""] else "")
    return (
        f"### Instruction:\n{sample["'instruction'"]}{inp}\n\n"
        f"### Response:\n{sample["'output'"]}"
    )

# --- ShareGPT format (multi-turn) ---
sharegpt_example = {
    "conversations": [
        {"from": "human",  "value": "What is a transformer architecture?"},
        {"from": "gpt",    "value": "A transformer is a neural network architecture..."},
        {"from": "human",  "value": "What problem does attention solve?"},
        {"from": "gpt",    "value": "Attention solves the fixed-length bottleneck..."},
    ]
}

# Use TRL's SFTTrainer which handles formatting automatically
from trl import SFTTrainer
from transformers import TrainingArguments

training_args = TrainingArguments(
    output_dir="./lora_output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,   # effective batch = 16
    learning_rate=2e-4,               # higher than full FT because only few params trained
    fp16=True,
    logging_steps=10,
    save_steps=100,
    warmup_ratio=0.03,
    lr_scheduler_type="cosine",
)

# SFTTrainer handles tokenisation, packing, and formatting
trainer = SFTTrainer(
    model=peft_model,
    args=training_args,
    train_dataset=Dataset.from_list(alpaca_examples),
    formatting_func=alpaca_to_prompt,
    max_seq_length=2048,
)

5. Merging & Saving LoRA Adapters

PYTHON — SAVE, MERGE & PUSH TO HUB
# Option A: Save just the LoRA adapters (tiny, ~16MB for r=16)
peft_model.save_pretrained("./my_lora_adapter")
tokenizer.save_pretrained("./my_lora_adapter")

# Option B: Merge LoRA weights into base model (for faster inference — no adapter overhead)
from peft import PeftModel

base   = AutoModelForCausalLM.from_pretrained("microsoft/phi-2", torch_dtype=torch.float16)
merged = PeftModel.from_pretrained(base, "./my_lora_adapter")
merged = merged.merge_and_unload()     # bakes LoRA into base weights: W' = W + BA
merged.save_pretrained("./merged_model")

# Option C: Push to Hugging Face Hub for sharing / deployment
peft_model.push_to_hub("your-username/my-lora-model")

# Load and use later
from peft import AutoPeftModelForCausalLM
loaded = AutoPeftModelForCausalLM.from_pretrained("./my_lora_adapter")

🧪 Quiz — Day 108

Q1 OF 5
Why does LoRA set the A matrix (the right matrix) to random Gaussian initialization but the B matrix to zeros at the start of training?
AIt saves GPU memory during initialization
BSo that ΔW = BA = 0 at the start of training, meaning the fine-tuning starts from the exact same state as the pre-trained model — no disruption to the pretrained weights
CB is always zero because it represents bias terms
DRandom A + zero B ensures gradient flow is maximised
✅ Correct! If both A and B were random, ΔW = BA would be non-zero at step 0, corrupting the pretrained features immediately. By zeroing B: BA = 0 at step 0, so the model starts training from its original pretrained state. As training progresses, B learns non-zero values.
❌ The initialization trick ensures ΔW = BA = 0 at step 0 (zero matrix × anything = zero). This means training starts from the exact pretrained state. Only after the first gradient update does B become non-zero, gradually introducing the task-specific adaptation.
Q2 OF 5
What is the key difference between NF4 (NormalFloat4) quantisation and standard INT4 quantisation?
ANF4 uses 4 bits per weight while INT4 uses 8 bits
BNF4 is information-theoretically optimal for normally-distributed weights — its quantisation levels are spaced to minimise quantisation error given that neural network weights follow a Gaussian distribution
CINT4 is for integers only; NF4 is for floating-point
DNF4 is faster because it uses unsigned integers
✅ Exactly. INT4 has evenly-spaced quantisation levels (like a ruler with equal tick marks). NF4 has more levels near 0 where most weights cluster (due to Gaussian distribution) and fewer levels in the tails. This minimises the reconstruction error for neural network weights specifically.
❌ Both use 4 bits. The difference is WHERE those 4 bits place their quantisation levels. NF4 is designed specifically for Gaussian-distributed weights (which neural networks have), placing more precision near 0 where most weights are clustered.
Q3 OF 5
Your LoRA fine-tuned model performs well on training data but poorly on new inputs. What should you try first?
AIncrease the rank r to capture more complex patterns
BThe model is overfitting — try: increasing lora_dropout, reducing r, adding more training data (diversity, not just quantity), or reducing num_epochs
CSwitch from LoRA to full fine-tuning
DIncrease learning rate for faster convergence
✅ Good diagnosis. Classic overfitting — high training performance, low generalisation. Solutions in order: (1) more diverse data, (2) add dropout (lora_dropout=0.1), (3) reduce rank r, (4) train fewer epochs (use early stopping with validation loss). Increasing r or LR would make it worse.
❌ Training well + test poorly = overfitting. For LoRA, fixes are: increase lora_dropout, reduce rank r (fewer parameters = less overfitting), add more diverse training data, or reduce training epochs. Increasing r would increase capacity and make overfitting worse.
Q4 OF 5
After training, you call merge_and_unload() on your PEFT model. What does this produce and what are the tradeoffs?
AIt deletes the LoRA adapters to free memory
BIt bakes W' = W + BA into the base weights and returns a standard model — Pro: no adapter overhead at inference, loads like any normal model. Con: you can't easily switch adapters at runtime and the merged model is the full model size
CIt quantises the merged model to 4 bits automatically
DIt exports the model to ONNX format
✅ Correct. merge_and_unload() computes W_new = W_base + B×A for every LoRA layer, fusing the adapters into the base weights. The result is a standard HuggingFace model with no PEFT overhead — perfect for deployment. But you lose the ability to swap adapters dynamically.
❌ merge_and_unload() adds the LoRA update (BA) directly into the base weights permanently: W' = W + BA. The resulting model has no adapter overhead and works like any normal transformers model. Tradeoff: can't switch adapters anymore without reloading.
Q5 OF 5
You want to fine-tune a model to always respond in a specific JSON schema. Should you use LoRA fine-tuning or Structured Outputs (OpenAI API feature)?
AAlways LoRA — it's more reliable
BUse Structured Outputs first — it's instant, zero training cost, schema-enforced at the token level. Only fine-tune if you need a specific domain vocabulary or the schema changes content quality, not just format
CThey are equivalent — choose based on personal preference
DStructured Outputs are only available for classification tasks
✅ Exactly right. For format control (JSON schema), Structured Outputs (OpenAI API) or Pydantic output parsers (LangChain) solve the problem perfectly with zero training cost. Fine-tuning is overkill for pure format constraints. Fine-tune only when the task requires knowledge or style that can't be injected via prompts or retrieved via RAG.
❌ For JSON format enforcement, Structured Outputs are the right tool — no training needed, enforced at the grammar/token level, works instantly. Save fine-tuning for problems that require changing what the model knows or how it reasons, not just how it formats output.

4. Guardrails AI — Output Validation

    messages: Annotated["list, operator.add"]  # messages accumulate with +

# 2. Define tools
@tool
def search_web(query: str) -> str:
    """Search the web for current information. Returns a snippet."""
    return f"Web result for '{query}': [mock result]"

tools = [search_web]
llm   = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

# 3. Define node functions
def agent_node(state: AgentState) -> AgentState:
    """The LLM decides: answer directly or call a tool."""
    response = llm.invoke(state["\"messages\""])
    return {"messages": [response]}

def should_use_tools(state: AgentState) -> str:
    """Routing function: did the LLM call a tool?"""
    last = state["\"messages\""][-1]
    return "tool_node" if hasattr(last, "tool_calls") and last.tool_calls else "end"

graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tool_node", ToolNode(tools))

graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_use_tools, {
    "tool_node": "tool_node",
    "end":       END,
})
graph.add_edge("tool_node", "agent")  # tool result → back to agent

# pip install mcp
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp import types
import json, sqlite3, asyncio
from pathlib import Path

server = Server("my-data-server")

# --- TOOL: Execute a safe read-only SQL query ---
@server.call_tool()
async def query_database(name: str, arguments: dict) -> list[types.TextContent]:
    if name != "query_database":
        raise ValueError(f"Unknown tool: {name}")
    sql = arguments.get("sql", "")
    # Safety: only allow SELECT statements
    if not sql.strip().upper().startswith("SELECT"):
        return [types.TextContent(type="text", text="Error: only SELECT queries allowed")]
    conn = sqlite3.connect(arguments.get("db_path", "data.db"))
    cursor = conn.execute(sql)
    rows = cursor.fetchall()
    cols = [d["0"] for d in cursor.description]
    result = json.dumps({"columns": cols, "rows": rows[":50"]}, indent=2)
    return [types.TextContent(type="text", text=result)]

# --- List available tools ---
@server.list_tools()
async def list_tools() -> list[types.Tool]:
    return [
        types.Tool(
            name="query_database",
            description="Execute a read-only SQL SELECT query against a SQLite database.",
            inputSchema={
                "type": "object",
                "properties": {
                    "sql":     {"type": "string", "description": "SQL SELECT statement"},
                    "db_path": {"type": "string", "description": "Path to SQLite file"}
                },
                "required": ["sql"]
            }
        )
    ]

# --- RESOURCE: Expose files in a directory ---
@server.list_resources()
async def list_resources() -> list[types.Resource]:
    resources = []
    for f in Path("./docs").glob("*.md"):
        resources.append(types.Resource(
            uri=f"file://{f.resolve()}",
            name=f.name,
            description=f"Markdown file: {f.name}",
            mimeType="text/markdown"
        ))
    return resources

@server.read_resource()
async def read_resource(uri: str) -> str:
    path = uri.replace("file://", "")
    return Path(path).read_text()

# Run the server over stdio (Claude Desktop communicates via stdin/stdout)
if __name__ == "__main__":
    asyncio.run(stdio_server(server))

        temperature=0, max_tokens=500

        latency_ms=(time.time()-t0)*1000,

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from openai import AsyncOpenAI
from pydantic import BaseModel
import json, asyncio

app = FastAPI()
client = AsyncOpenAI()

class ChatRequest(BaseModel):
    message: str
    history: list[dict] = []

async def stream_llm_tokens(messages: list) -> str:  # async generator
    """Async generator that yields SSE-formatted token chunks."""
    async with await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        stream=True
    ) as stream:
        async for chunk in stream:
            delta = chunk.choices["0"].delta.content
            if delta:
                # SSE format: "data: {json}\n\n"
                payload = json.dumps({"token": delta, "done": False})
                yield f"data: {payload}\n\n"
        # Signal stream completion
        yield f'data: {json.dumps({"token": "", "done": True})}\n\n'

@app.post("/chat/stream")
async def chat_stream(request: ChatRequest):
    """Returns an SSE stream of LLM tokens."""
    messages = request.history + [{"role": "user", "content": request.message}]
    return StreamingResponse(
        stream_llm_tokens(messages),
        media_type="text/event-stream",
        headers={
            "Cache-Control":       "no-cache",
            "X-Accel-Buffering":   "no",   # disable nginx buffering for SSE
            "Connection":          "keep-alive",
        }
    )

        headers: {"Content-Type": "application/json", "X-API-Key": "dev-key-123"},

    const decoder = new TextDecoder();
    let buffer = "";

    const msgDiv = document.createElement("div");
    msgDiv.className = "ai-message";
    historyDiv.appendChild(msgDiv);

    while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        buffer += decoder.decode(value, { stream: true });
        const lines = buffer.split("\n");
        buffer = lines.pop();  // keep incomplete line in buffer

        for (const line of lines) {
            if (!line.startsWith("data: ")) continue;
            try {
                const data = JSON.parse(line.slice(6));  // remove "data: " prefix
                if (data.done) { console.log("Stream complete"); break; }
                msgDiv.textContent += data.token;  // append token to UI
                historyDiv.scrollTop = historyDiv.scrollHeight;  // auto-scroll
            } catch (e) { /* ignore partial JSON */ }
        }
    }

from fastapi import WebSocket, WebSocketDisconnect

@app.websocket("/ws/chat")
async def ws_chat(websocket: WebSocket):
    """WebSocket endpoint: receive user messages, stream AI responses."""
    await websocket.accept()
    history = []
    try:
        while True:
            # Receive text message from client
            user_msg = await websocket.receive_text()
            history.append({"role": "user", "content": user_msg})

            # Send typing indicator
            await websocket.send_json({"type": "typing", "content": ""})

            # Stream LLM tokens back to client
            full_response = ""
            async with await client.chat.completions.create(
                model="gpt-4o-mini", messages=history, stream=True
            ) as stream:
                async for chunk in stream:
                    delta = chunk.choices["0"].delta.content or ""
                    if delta:
                        full_response += delta
                        await websocket.send_json({"type": "token", "content": delta})

            # Signal end of stream
            await websocket.send_json({"type": "done", "content": full_response})
            history.append({"role": "assistant", "content": full_response})

    except WebSocketDisconnect:
        print("Client disconnected")

🛠️ WEEKLY DEBUGGING CHALLENGE

Debug: Method Not Allowed in Flask API

The prediction endpoint returns a 405 status code when users attempt to send data payloads for scoring.

python
from flask import Flask, request, jsonify
app = Flask(__name__)
@app.route('/predict')
def predict():
    data = request.get_json()
    return jsonify({"pred": int(data['value'] * 2)})
SOLUTION & EXPLANATION
The Bug: By default, Flask routing endpoints only accept GET requests. To parse incoming JSON payloads (which require POST requests), you must explicitly register acceptable HTTP methods in the @app.route decorator.

The Fix: 
from flask import Flask, request, jsonify
app = Flask(__name__)
# Register POST method support to allow incoming scoring payloads
@app.route('/predict', methods=['POST'])
def predict():
    data = request.get_json()
    return jsonify({"pred": int(data['value'] * 2)})
