🧠 Theory

1. Flask Architecture — WSGI & the Request-Response Cycle

Flask is a WSGI (Web Server Gateway Interface) application. Every HTTP request flows: Browser → Web Server (nginx/gunicorn) → WSGI App (Flask) → Your code → Response. Flask wraps this complexity into a clean decorator-based API.

ConceptWhat It IsFlask Object
RouteURL pattern mapped to a Python function@app.route('/path')
View FunctionPython function that handles a requestAny function under @app.route
Request ContextThread-local object holding current request dataflask.request
Application ContextHolds app-level data (config, db connections)flask.g, flask.current_app
ResponseWhat Flask returns to the clientflask.Response, jsonify(), string
PYTHON — MINIMAL FLASK APP
# pip install flask
from flask import Flask, request, jsonify

app = Flask(__name__)   # __name__ tells Flask where to find templates/static files

# Basic GET route
@app.route('/')
def index():
    return 'Hello, AI/ML World!'

# Route with URL parameter
@app.route('/model/<string:model_name>')
def model_info(model_name: str):
    return jsonify({'model': model_name, 'status': 'active'})

# POST route that reads JSON body
@app.route('/predict', methods=['POST'])
def predict():
    data = request.get_json()           # parse JSON body
    if data is None:
        return jsonify({'error': 'No JSON body'}), 400
    features = data.get('features', [])
    return jsonify({'prediction': 'placeholder', 'input_len': len(features)})

# Query string: GET /search?q=neural&limit=5
@app.route('/search')
def search():
    query = request.args.get('q', '')
    limit = request.args.get('limit', 10, type=int)
    return jsonify({'query': query, 'limit': limit})

if __name__ == '__main__':
    app.run(debug=True, host='0.0.0.0', port=5000)
Flask ko samjho jaise ek post office counter. Ek request aata hai (customer), route (@app.route) decide karta hai kaun serve karega (kaunsa clerk), view function (clerk) kaam karta hai, aur response wapas jata hai (parcel deliver). Bina route ke, request lost ho jaata hai.

2. Application Factory Pattern

Production apps never create the Flask app at module level — instead they use a factory function. This enables testing (create fresh app per test), multiple configs (dev/prod/test), and avoids circular imports.

PYTHON — APPLICATION FACTORY (app/__init__.py)
# Folder structure:
# myapp/
#   __init__.py      ← app factory here
#   config.py        ← config classes
#   routes/
#     __init__.py
#     predict.py     ← prediction blueprint
#     health.py      ← health check blueprint

from flask import Flask
from .config import DevelopmentConfig, ProductionConfig
import os

def create_app(config_name: str = None) -> Flask:
    """Application factory — call this to get a configured Flask app."""
    app = Flask(__name__)

    # Load config from environment or explicit name
    cfg_map = {'dev': DevelopmentConfig, 'prod': ProductionConfig}
    cfg_name = config_name or os.environ.get('FLASK_ENV', 'dev')
    app.config.from_object(cfg_map.get(cfg_name, DevelopmentConfig))

    # Register blueprints
    from .routes.predict import predict_bp
    from .routes.health import health_bp
    app.register_blueprint(predict_bp, url_prefix='/api/v1')
    app.register_blueprint(health_bp)

    return app
PYTHON — BLUEPRINT (routes/predict.py)
from flask import Blueprint, request, jsonify

predict_bp = Blueprint('predict', __name__)   # Blueprint name + module

@predict_bp.route('/predict', methods=['POST'])
def predict():
    data = request.get_json(force=True, silent=True)
    if not data:
        return jsonify({'error': 'Invalid JSON'}), 400
    # model inference goes here (Day 120)
    return jsonify({'prediction': 0, 'confidence': 0.95})

# Run: flask --app "myapp:create_app()" run

3. Error Handling & HTTP Status Codes

Good APIs return meaningful HTTP status codes. Use Flask's abort() for standard errors, or register custom error handlers.

PYTHON — ERROR HANDLERS
from flask import jsonify
from werkzeug.exceptions import HTTPException

def register_error_handlers(app):
    @app.errorhandler(400)
    def bad_request(e):
        return jsonify({'error': 'Bad Request', 'detail': str(e)}), 400

    @app.errorhandler(404)
    def not_found(e):
        return jsonify({'error': 'Endpoint not found'}), 404

    @app.errorhandler(422)
    def validation_error(e):
        return jsonify({'error': 'Unprocessable Entity', 'detail': str(e)}), 422

    @app.errorhandler(500)
    def server_error(e):
        return jsonify({'error': 'Internal Server Error'}), 500

    # Catch all HTTP exceptions uniformly
    @app.errorhandler(HTTPException)
    def handle_http_exception(e):
        return jsonify({'error': e.name, 'code': e.code}), e.code
💡 Flask vs FastAPI: Flask is synchronous by default and battle-tested (since 2010). FastAPI (which you used in Week 16) is async-first and auto-generates OpenAPI docs. For CPU-bound ML serving, Flask + Gunicorn workers is completely production-viable. For async IO-heavy workloads (streaming, concurrent requests), FastAPI is better. Both are right tools — know when to use which.

🧪 Quiz — Day 118

Q1 OF 5
What does Flask's debug=True mode do, and why must you NEVER use it in production?
AIt disables all logging to speed up responses
BIt enables auto-reload on code changes AND exposes an interactive debugger in the browser — which gives anyone who can reach the server a Python shell with full server access
CIt only enables auto-reload with no security implications
DIt enables HTTPS automatically
✅ Correct! The Werkzeug debugger is a full Python REPL accessible in the browser when an exception occurs. In production, anyone who can trigger an error gets code execution on your server. Always set debug=False and use FLASK_ENV=production in prod.
❌ debug=True enables two things: (1) auto-reloader on file changes, (2) the Werkzeug interactive debugger shown in browser on errors. The debugger is effectively a Python shell — a critical security vulnerability if exposed publicly.
Q2 OF 5
You call request.get_json() and get None back. What are the two most likely causes?
AThe request body is empty AND the server is in debug mode
BThe Content-Type header is not set to application/json, OR the body is malformed JSON (not valid JSON syntax)
CThe route method is GET, not POST
DFlask version is outdated
✅ Exactly. Use get_json(force=True) to skip Content-Type check, and get_json(silent=True) to suppress parse errors (returns None instead of 400). In production, always handle None explicitly rather than relying on silent=True masking bugs.
❌ get_json() returns None when: (1) Content-Type != application/json (unless force=True), (2) the body is not valid JSON. Fix with: request.get_json(force=True, silent=True) and check for None explicitly.
Q3 OF 5
What is the main benefit of using Blueprints over putting all routes in a single app.py?
ABlueprints make Flask run faster by pre-compiling routes
BBlueprints allow modular code organisation — each feature has its own file with its own URL prefix, and can be registered/unregistered without touching other code
CBlueprints add automatic authentication to routes
DBlueprints are mandatory for any Flask app with more than 5 routes
✅ Right. Blueprints are Flask's module system. Each blueprint has its own routes, error handlers, templates, and static files. You can mount the same blueprint at different URL prefixes, enabling API versioning (register predict_bp at /api/v1 and /api/v2 with different handlers).
❌ Blueprints are about code organisation and maintainability. They let you split a large app into smaller, self-contained modules. Each blueprint can have its own routes, templates, and URL prefix — crucial for team development and API versioning.
Q4 OF 5
What HTTP status code should a POST /predict endpoint return when the request body is syntactically valid JSON but the features field contains strings instead of floats?
A400 Bad Request
B422 Unprocessable Entity — the JSON is syntactically valid but semantically wrong (wrong field type)
C500 Internal Server Error
D200 OK with an error field in the response body
✅ Correct. 400 = malformed request (can't parse). 422 = well-formed but semantically invalid (correct JSON, wrong values). For ML APIs: 400 when JSON can't be parsed, 422 when JSON is valid but field types/values fail schema validation.
❌ The distinction: 400 Bad Request = JSON syntax error (can't parse at all). 422 Unprocessable Entity = JSON parsed fine but field validation failed (wrong type, out-of-range value). FastAPI and modern REST APIs use 422 for schema violations.
Q5 OF 5
Why is Flask's built-in development server (flask run) not suitable for production?
AIt doesn't support Python 3
BIt's single-threaded (one request at a time), not designed for load, has no graceful shutdown, and Werkzeug's warning itself says not to use it in production
CIt can't serve JSON responses
DIt requires root privileges
✅ Exactly. Use Gunicorn (gunicorn -w 4 "myapp:create_app()") or uWSGI in production. Gunicorn spawns multiple worker processes, handles graceful restarts, and can sit behind nginx for SSL termination and static file serving.
❌ Flask's dev server is single-threaded and not optimised for concurrency or production load. The fix: pip install gunicorn and run gunicorn -w 4 -b 0.0.0.0:5000 "myapp:create_app()" — 4 worker processes, each handling requests independently.

Fix: Scale workers using Gunicorn process managers, or configure reverse-proxy load balancers to distribute traffic.

1. REST Resource Design for ML APIs

REST treats every "thing" as a resource with a URL. For ML APIs, your resources are typically: predictions, models, and jobs (async tasks). URLs should be nouns, not verbs.

/api/v1/predictPOSTSynchronous prediction200

2. Consistent JSON Response Envelope

PYTHON — RESPONSE ENVELOPE PATTERN
from flask import jsonify
import time, uuid

def success_response(data, message="OK", status_code=200):
    """Standard success envelope for all ML API responses."""
    return jsonify({
        "success": True,
        "message": message,
        "data": data,
        "request_id": str(uuid.uuid4()),
        "timestamp": time.time()
    }), status_code

def error_response(error, detail=None, status_code=400):
    """Standard error envelope — consistent shape for all errors."""
    return jsonify({
        "success": False,
        "error": error,
        "detail": detail,
        "request_id": str(uuid.uuid4()),
        "timestamp": time.time()
    }), status_code

# Usage in a view function:
@predict_bp.route('/predict', methods=['POST'])
def predict():
    data = request.get_json(force=True, silent=True)
    if not data:
        return error_response("Invalid JSON", status_code=400)
    # ... model inference ...
    return success_response({
        "prediction": 1,
        "confidence": 0.87,
        "model_version": "v2.1.0"
    })

3. Logging Middleware & Rate Limiting

PYTHON — MIDDLEWARE + RATE LIMITING
from flask import Flask, request, g
from flask_limiter import Limiter
from flask_limiter.util import get_remote_address
import logging, time, uuid

logger = logging.getLogger(__name__)
limiter = Limiter(key_func=get_remote_address, default_limits=["100/hour"])

def create_app():
    app = Flask(__name__)
    limiter.init_app(app)

    @app.before_request
    def before():
        g.start_time = time.time()
        g.request_id = str(uuid.uuid4())[:8]
        logger.info(f"[{g.request_id}] {request.method} {request.path}")

    @app.after_request
    def after(response):
        duration_ms = (time.time() - g.start_time) * 1000
        response.headers["\"X-Request-ID\""] = g.request_id
        response.headers["\"X-Response-Time\""] = f"{duration_ms:.1f}ms"
        logger.info(f"[{g.request_id}] {response.status_code} {duration_ms:.1f}ms")
        return response

    # Apply tight rate limit to expensive predict endpoint
    @app.route("/api/v1/predict", methods=["POST"])
    @limiter.limit("10/minute")
    def predict():
        return {"prediction": 0}

    return app
🔑 request_id Tracing: Always generate a unique request ID per request and attach it to every log line and the response header (X-Request-ID). When a client reports a bug, they give you the request ID, and you can grep logs across services to reconstruct the full trace.

from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline

# --- TRAINING TIME (run once, save artifacts) ---
pipeline = Pipeline([

PYTHON — FLASK APP WITH MODEL LOADED AT STARTUP
from flask import Flask, request, jsonify
import joblib, numpy as np, os

# MODULE-LEVEL: model loaded once when the process starts
# With Gunicorn -w 4, this runs 4 times (once per worker) — that's fine.
MODEL_PATH = os.environ.get('MODEL_PATH', 'models/iris_pipeline_v1.joblib')
META_PATH  = os.environ.get('META_PATH',  'models/iris_metadata_v1.joblib')

pipeline = None
metadata = None

def create_app():
    app = Flask(__name__)

    global pipeline, metadata
    try:
        pipeline = joblib.load(MODEL_PATH)
        metadata = joblib.load(META_PATH)
        app.logger.info(f"Model v{metadata["'version'"]} loaded from {MODEL_PATH}")
    except FileNotFoundError as e:
        app.logger.error(f"Model file not found: {e}")
        # Don't crash — let /health/ready return 503 instead

    @app.route('/health/ready')
    def ready():
        if pipeline is None:
            return jsonify({'status': 'not_ready', 'reason': 'model not loaded'}), 503
        return jsonify({'status': 'ready', 'model_version': metadata["'version'"]})

    @app.route('/api/v1/predict', methods=['POST'])
    def predict():
        if pipeline is None:
            return jsonify({'error': 'Model not loaded'}), 503

        data = request.get_json(force=True, silent=True)
        if not data or 'features' not in data:
            return jsonify({'error': 'Missing features field'}), 400

        try:
            X = np.array(data["'features'"]).reshape(1, -1)
            pred  = pipeline.predict(X)[0]
            proba = pipeline.predict_proba(X)[0].tolist()
            return jsonify({
                'prediction': int(pred),
                'class_name': metadata["'classes'"][pred],
                'probabilities': {k: round(v, 4) for k, v in zip(metadata["'classes'"], proba)},
                'model_version': metadata["'version'"]
            })
        except (ValueError, TypeError) as e:
            return jsonify({'error': 'Invalid features', 'detail': str(e)}), 422

    return app

# Production: gunicorn -w 4 -b 0.0.0.0:5000 "app:create_app()"
⚠️ Concurrency Warning: sklearn's predict() releases the GIL — Gunicorn sync workers are safe for sklearn. But PyTorch models are NOT thread-safe by default. For PyTorch in production, either: (1) use Gunicorn with --worker-class gthread and a lock, (2) use TorchServe, (3) deploy as an async service with a request queue.

2. Gunicorn Configuration

BASH — GUNICORN PRODUCTION COMMAND
# Rule of thumb: workers = (2 × CPU_cores) + 1
gunicorn \
  --workers 4 \
  --worker-class sync \          # sync = blocking, suitable for CPU-bound ML inference

🧠 Theory

1. Layer Caching — The Key to Fast Builds

Docker caches each layer. If a layer hasn't changed, Docker reuses the cache for that layer AND all subsequent layers. The trick: put things that change rarely (dependencies) before things that change often (source code).

DOCKERFILE — BAD (no caching optimisation)
# ❌ BAD: COPY . . before pip install means every code change
# forces re-installing ALL packages — even if requirements.txt didn't change
FROM python:3.11-slim
WORKDIR /app
COPY . .                         # copies everything first
RUN pip install -r requirements.txt  # cache invalidated on ANY file change!
CMD ["gunicorn", "app:create_app()"]
DOCKERFILE — GOOD (layer cache optimised)
# ✅ GOOD: requirements.txt first → pip install cached unless deps change
FROM python:3.11-slim

# Install system deps (rarely changes → cached almost always)
RUN apt-get update && apt-get install -y --no-install-recommends \
    libgomp1 \                  # needed by scikit-learn
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Copy ONLY requirements first (changes rarely)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy model artifacts (changes occasionally)
COPY models/ ./models/

# Copy source code LAST (changes most often)
COPY . .

# Security: run as non-root user
RUN useradd --create-home --shell /bin/bash appuser
USER appuser

EXPOSE 5000
CMD ["gunicorn", "--workers", "4", "--bind", "0.0.0.0:5000", "--timeout", "120", "app:create_app()"]

2. Multi-Stage Builds for ML Images

DOCKERFILE — MULTI-STAGE (small prod image)
# Stage 1: Builder — install deps + compile anything needed
FROM python:3.11-slim AS builder
WORKDIR /build
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt
# --user installs to ~/.local, easy to copy to runtime stage

# Stage 2: Runtime — only what's needed to run
FROM python:3.11-slim AS runtime
WORKDIR /app

# Copy installed packages from builder stage
COPY --from=builder /root/.local /root/.local

# Copy app source and model
COPY app/ ./app/
COPY models/ ./models/

# Non-root user
RUN useradd -m appuser && chown -R appuser:appuser /app
USER appuser

# Put user packages on PATH
ENV PATH=/root/.local/bin:$PATH

EXPOSE 5000
CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:5000", "app:create_app()"]

# Build: docker build --target runtime -t my-ml-app:v1 .
# Result: ~200MB instead of ~800MB for the same app

2. Prediction Caching with Redis

PYTHON — REDIS CACHE IN FLASK
import redis, json, hashlib, os

redis_client = redis.from_url(
    os.environ.get('REDIS_URL', 'redis://localhost:6379/0'),
    decode_responses=True
)

def cache_key(features: list) -> str:
    """Create deterministic cache key from feature vector."""
    payload = json.dumps(sorted(features), separators=(",", ":"))
    return f"pred:{hashlib.sha256(payload.encode()).hexdigest()[:16]}"

@predict_bp.route('/predict', methods=['POST'])
def predict():
    data = request.get_json(force=True, silent=True)
    if not data:
        return jsonify({'error': 'Invalid JSON'}), 400

    features = data.get('features', [])
    key = cache_key(features)

    # Cache hit: return immediately without running model
    cached = redis_client.get(key)
    if cached:
        result = json.loads(cached)
        result["'cache'"] = 'hit'
        return jsonify(result)

    # Cache miss: run model inference
    X = np.array(features).reshape(1, -1)
    pred  = pipeline.predict(X)[0]
    proba = pipeline.predict_proba(X)[0].tolist()
    result = {
        'prediction': int(pred),
        'class_name': metadata["'classes'"][pred],
        'probabilities': proba,
        'cache': 'miss'
    }

    # Store in cache for 1 hour (TTL=3600)
    redis_client.setex(key, 3600, json.dumps(result))
    return jsonify(result)

🧪 Week 17 Final Quiz

Q1 OF 5
In a Flask + Gunicorn setup with 4 workers, where should you load your ML model to avoid loading it on every request?
AInside the view function, using a local variable
BAt module level or in create_app() — loaded once per worker process when Gunicorn forks, then reused across all requests handled by that worker
CIn a before_request hook so it loads fresh each time
DIn a separate thread that shares the model via a global queue
✅ Correct. With Gunicorn -w 4, there are 4 worker processes. Each loads the model once at startup. All requests to that worker reuse the loaded model. Loading in the view function would load the model on every single request — potentially seconds of latency each time.
❌ Load the model at module level or in create_app(). Gunicorn workers are forked after app initialisation, so each worker inherits the loaded model. Loading per-request = O(seconds) latency overhead = unacceptable in production.
Q2 OF 5
Why does the Flask app in docker-compose NOT expose port 5000 to the host, while Nginx DOES expose port 80?
ADefence in depth — Flask should only be accessible through Nginx, not directly from the internet. Nginx handles SSL, rate limiting, and acts as a single entry point. If Flask were exposed, clients could bypass Nginx's protections.
BDocker requires one container to own each port
CFlask can't bind to external interfaces
DPort 5000 is reserved by the OS
✅ Exactly. The pattern is: expose only the reverse proxy (Nginx on 80/443), keep app servers (Gunicorn) and data stores (Redis) internal to the Docker network. This is the standard "proxy + app + data" separation used in production. Only Nginx needs to be internet-facing.
❌ It's a security architecture decision. Only Nginx should be internet-accessible. Flask runs on the internal Docker network — Nginx proxies to it by container name. This prevents clients from bypassing Nginx (and its rate limiting, SSL termination, etc.) by hitting Flask directly.
Q3 OF 5
You changed one line in app.py. With proper Dockerfile layer ordering, which layers need to be rebuilt?
AAll layers from scratch including pip install
BOnly the "COPY . ." layer and everything after it. The base image, apt-get, and pip install layers are cached and reused.
COnly the CMD layer changes
DThe entire image is always rebuilt from scratch
✅ Right. Order: FROM (cache hit) → RUN apt-get (cache hit) → COPY requirements.txt (cache hit) → RUN pip install (cache hit) → COPY models/ (cache hit) → COPY . . (CACHE MISS — app.py changed) → USER/EXPOSE/CMD (rebuilt). Result: rebuild in ~5 seconds instead of ~3 minutes.
❌ With correct layer ordering (requirements first, source last), changing app.py only invalidates the COPY . . layer and everything after. Base image, apt-get, pip install all hit the cache. That's why layer ordering is so important — it determines rebuild time.
Q4 OF 5
Your Flask /predict endpoint receives {"features": "5.1,3.5,1.4,0.2"} (a string, not a list). What HTTP status code and response should you return?
A200 OK — try to parse the string automatically
B422 Unprocessable Entity — valid JSON but wrong type for the features field; return field-level error: {"error": "Validation failed", "detail": {"features": "must be a list of numbers"}}
C400 Bad Request — the JSON is malformed
D500 Internal Server Error — let numpy handle the bad input
✅ Correct. The JSON is syntactically valid (a string is legal JSON). The problem is semantic — the field type is wrong. 422 is the correct code for semantic validation failures. Never let bad input reach numpy/sklearn — always validate input types and shapes at the API boundary.
❌ The JSON is valid — it's {"features": "5.1,3.5,1.4,0.2"} which parses fine. But the value is a string, not a list. That's a 422 (semantic error). 400 = syntax/parse error. 500 = never intentionally return this from bad user input. Always validate at the API boundary.
Q5 OF 5
What is the purpose of depends_on: condition: service_healthy in docker-compose, and what must you define on the dependency for it to work?
AIt's the same as depends_on with no condition — just a newer syntax
BIt delays the dependent service from starting until the dependency passes its healthcheck. The dependency service must have a healthcheck: block defined, otherwise Compose can't know when it's "healthy"
CIt automatically restarts the dependent service if the dependency crashes
DIt requires Docker Swarm mode to work
✅ Exactly. Without condition: service_healthy, depends_on only waits for the container to START (not to be ready). Redis might be starting but not ready to accept connections when Flask boots. The healthcheck block (test: redis-cli ping) defines what "healthy" means. Compose polls it until it passes, then starts Flask.
❌ depends_on without condition only waits for the container process to start — not for the service inside to be ready. Redis takes ~1 second to initialise. condition: service_healthy polls the healthcheck (redis-cli ping) until it returns exit 0, then starts the dependent service. Requires a healthcheck block on the dependency.

🛠️ WEEKLY DEBUGGING CHALLENGE

Debug: Database Connection Refused in Compose Container

A containerized API fails to connect to the database container, throwing a connection refused exception.

python
# Configuration inside compose container trying to reach database service
DB_URL = "postgresql://user:pass@127.0.0.1:5432/dbname"
SOLUTION & EXPLANATION
The Bug: Each container in Docker runs on its own isolated loopback network. Using 127.0.0.1 inside a container resolves to that container itself, not the host machine or other containers. In Docker Compose networks, containers communicate using service names as hostname aliases.

The Fix: 
# Connect using the compose service name ('db') instead of localhost
DB_URL = "postgresql://user:pass@db:5432/dbname"
