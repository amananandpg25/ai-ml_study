1. Why Pandas? The Data Science Workhorse

Excel toh sab jaante hain — rows aur columns mein data. Pandas bilkul wahi hai, lekin code se control karo. Ek click ki jagah ek line Python — aur millions of rows handle karo bina hangs ke. Bade companies ka data terabytes mein hota hai — Excel toh crash ho jata hai, Pandas nahi.

Pandas has two core data structures: Series (1D — like a single column) and DataFrame (2D — like a full spreadsheet). Almost all ML workflows start with loading a DataFrame from a CSV file.

In ML, your dataset IS a DataFrame. Features (columns) go into X, target (label column) goes into y. Understanding DataFrame operations like iloc, loc, and boolean indexing is essential before you can split data for training.

⚡ Vectorization vs Loops: Why is Pandas so fast? Under the hood, Pandas uses NumPy (C/C++). Never use for loops to iterate through a DataFrame if you can avoid it. Vectorized operations (like df["A"] + df["B"]) apply to the entire column at once, running 100x-1000x faster than a Python loop.
🚀 Scaling Mindset & Advanced Curiosity: Pandas is great for "medium" data (fits in RAM, up to a few GBs). What happens at scale? Pandas struggles with huge data because it loads everything into memory. For terabytes of data, companies use distributed engines like Spark or out-of-core tools like Dask and Polars.

⚠️ iloc vs loc — The #1 Confusion: iloc uses integer positions (like list indexing): df.iloc[0] = first row regardless of index.
loc uses labels: df.loc[0, "name"] — works by index label AND column name.
If your DataFrame has a custom index like ["a","b","c"], df.iloc[0] = row "a" but df.loc[0] = KeyError! Always know your index.

9. Slicing Caution: SettingWithCopyWarning & .copy()

When you slice a subset of a DataFrame (e.g. subset = df[df["salary"] > 55000]) and then try to modify it (e.g. subset["bonus"] = 100), Pandas will frequently throw a SettingWithCopyWarning. This happens because Pandas is unsure whether your slice is a View (points to the original data) or a Copy (a separate object in memory). Modifying a view modifies the original, while modifying a copy does not.

⚠️ SettingWithCopyWarning is NOT a crash: It is a warning to prevent you from writing code that might fail to update the original DataFrame or update it accidentally.

How to fix it: If you want a completely independent copy to modify safely, explicitly use .copy():

python — safe_copy.py
# Safe way to modify a sliced subset without warnings
high_earners = df[df["salary"] > 55000].copy()
high_earners["bonus"] = high_earners["salary"] * 0.1  # Safe! No warning.

# If you WANT to modify the original DataFrame, use .loc:
df.loc[df["salary"] > 55000, "bonus"] = df["salary"] * 0.1  # Safe and direct!

# contact_book.py — Python Review Warm-up
import csv, os

class Contact:
    def __init__(self, name, phone, email, city):
        self.name = name.strip().title()
        self.phone = phone.strip()
        self.email = email.strip().lower()
        self.city = city.strip().title()

    def __str__(self):
        return f"{self.name:<18} | {self.phone:<12} | {self.email:<23} | {self.city}"

    def to_dict(self):
        return {"name":self.name,"phone":self.phone,"email":self.email,"city":self.city}

class ContactBook:
    FIELDS = ["name","phone","email","city"]

    def __init__(self, filename="contacts.csv"):
        self.filename = filename
        self.contacts = []
        self.load()

    def load(self):
        if not os.path.exists(self.filename): return
        try:
            with open(self.filename,"r") as f:
                for row in csv.DictReader(f):
                    self.contacts.append(Contact(row["name"],row["phone"],row["email"],row["city"]))
        except Exception as e:
            print(f"Load error: {e}")

    def save(self):
        with open(self.filename,"w",newline="") as f:
            w = csv.DictWriter(f,fieldnames=self.FIELDS)
            w.writeheader()
            w.writerows([c.to_dict() for c in self.contacts])

    def add(self):
        name = input("Name: ")
        if any(c.name.lower()==name.lower() for c in self.contacts):
            print("Contact already exists!"); return
        self.contacts.append(Contact(name, input("Phone: "), input("Email: "), input("City: ")))
        self.save(); print("✅ Contact added!")

    def search(self):
        q = input("Search (name/city): ").lower()
        results = [c for c in self.contacts if q in c.name.lower() or q in c.city.lower()]
        for c in results: print(c)

    def delete(self):
        name = input("Delete contact name: ")
        for i, c in enumerate(self.contacts):
            if c.name.lower() == name.lower():
                self.contacts.pop(i)
                self.save(); print("✅ Deleted!"); return
        print("Contact not found!")

    def list_all(self):
        if not self.contacts: print("No contacts yet!"); return
        for c in sorted(self.contacts, key=lambda c: c.name): print(c)

def main():
    book = ContactBook()
    menu = {"1":book.add,"2":book.search,"3":book.delete,"4":book.list_all}
    while True:
        print("\n📒 1.Add 2.Search 3.Delete 4.List 5.Exit")
        choice = input("Choice: ").strip()
        if choice == "5": break
        if choice in menu: menu[choice]()

if __name__ == "__main__":
    main()

import pandas as pd
import numpy as np

df = pd.DataFrame({
    "name":  ["Rahul", "Priya", np.nan, "Arjun"],
    "age":   [25, np.nan, 30, 22],
    "score": [85, 90, np.nan, np.nan]

# Detect missing values

print(df.isnull().sum())    # count of nulls per column
print(df.isnull().mean())   # percentage of nulls per column

df_clean = df.dropna()          # only Rahul row survives

# Drop rows only if SPECIFIC column is null
df2 = df.dropna(subset=["name"])  # drops only the np.nan name row

df["age"] = df["age"].fillna(0)      # Reassignment is best practice

# Fill with mean (better for numeric data)
mean_score = df["score"].mean()
df["score"] = df["score"].fillna(mean_score)  # fills 87.5

# Fill with forward fill (use previous row's value)

🧠 Theory

1. SQL vs. Pandas — The Paradigm Shift

Pandas is like cooking in your own small kitchen (RAM) where you pull all ingredients from the fridge (CSV) onto your counter and chop them up step-by-step (procedural). SQL is like ordering from a mega-warehouse: you tell the clerk exactly WHAT you want (declarative), and the warehouse engine fetches it from giant storage shelves (disk) using efficient forklifts.

As a data scientist, you already know Pandas. But Pandas has a limit: it loads the entire dataset into RAM. If you have a 50GB dataset and 16GB of RAM, Pandas will crash. SQL (Structured Query Language) solves this. SQL is declarative (you define what data you want, not how to get it) and operates directly on database files stored on disk, streaming only the necessary results to your computer.

3. Aggregations, GROUP BY & The HAVING Scaffold

Suppose you want to find statistics about departments. We use Aggregations (like SUM or AVG) to collapse multiple values into one. But how do we group them?

📐 The "Collapse" Concept of GROUP BY: When you run GROUP BY department, the database groups rows matching the same department into single summary rows. Any non-grouped column (like salary) must have an aggregate function applied to tell the engine how to shrink those rows (e.g., take the AVG or SUM).
sql — group_by_basics.sql
-- Simple Aggregation (Pandas: df['salary'].mean())
SELECT AVG(salary) AS avg_sal FROM employees;

-- Grouping rows (Pandas: df.groupby('department')['salary'].mean())
SELECT department, AVG(salary) AS avg_sal, COUNT(*) AS headcount
FROM employees
GROUP BY department;

The HAVING Limitation (Where vs Having)

What if you want to filter out departments with an average salary of less than 60,000? Your first instinct might be to write this:

sql — wrong_where.sql
-- ❌ THIS WILL FAIL WITH A SYNTAX ERROR!
SELECT department, AVG(salary)
FROM employees
WHERE AVG(salary) > 60000  -- Aggregates not allowed in WHERE
GROUP BY department;
⚠️ Why did it fail? (SQL Execution Order): The database engine runs the WHERE clause before it groups the data. Since grouping hasn't happened yet, the engine doesn't know what the average salary per department is! To filter groups, we must use HAVING, which is executed after GROUP BY.
sql — correct_having.sql
-- ✅ CORRECT: HAVING filters AFTER grouping
SELECT department, AVG(salary) AS avg_sal
FROM employees
GROUP BY department
HAVING AVG(salary) > 60000;

4. JOINs — Combining Tables from First Principles

When data is split across tables (e.g. employees and departments), we must join them. Let's understand how a join actually works step-by-step.

Step 1: The Cartesian Product (CROSS JOIN)

If you join two tables without any matching keys, the engine matches every row of Table A with every row of Table B. If Table A has 100 rows and Table B has 10 rows, the result is 1,000 rows. This is a Cartesian Product.

sql — cross_join.sql
SELECT e.name, d.department_name
FROM employees e
CROSS JOIN departments d;  -- Matches everything with everything

Step 2: INNER JOIN (Filtering the Cartesian Product)

To make the join meaningful, we add an ON clause. This acts like a filter on the Cartesian Product, retaining only rows where keys match (e.g. e.dept_id = d.id).

sql — inner_join.sql
SELECT e.name, d.department_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;  -- Only matching records kept

Step 3: LEFT JOIN (Handling Dangling Keys)

If an employee has a dept_id of NULL (or a department that doesn't exist), the `INNER JOIN` will drop that employee row. We call this a dangling key. To retain all employees regardless of whether they have a department match, we use a `LEFT JOIN`.

sql — left_join.sql
SELECT e.name, d.department_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;  -- Retains all employees, shows NULL if no match
🧠 Database Execution Order (The Execution Pipeline): To build complete query intuition, memorize the physical order in which a database processes your query:
FROM ➔ JOIN ➔ ON ➔ WHERE ➔ GROUP BY ➔ HAVING ➔ SELECT ➔ DISTINCT ➔ ORDER BY ➔ LIMIT
💡 Ek line mein: SELECT kya chahiye, FROM kahan se, WHERE kon sa row, GROUP BY kaise group karo, HAVING group filter, ORDER BY sort karo, LIMIT kitna lena. JOIN = do tables combine karo common key pe.

7. Database Normalization — 1NF, 2NF, 3NF

🧠 Why Normalization Matters: Normalization removes data redundancy and prevents update anomalies.
1NF — Each cell has one atomic value. No repeating groups. (No comma-separated lists in a column!)
2NF — 1NF + every non-key column depends on the whole primary key (not just part of it).
3NF — 2NF + no transitive dependencies (non-key column must depend only on the primary key, not on another non-key column).

Example: Storing "city" and "state" in an orders table where city already determines state — that's a 3NF violation. Move city→state mapping to a separate lookup table.

1. Subqueries — The "Stuck Query" Problem

Imagine trying to find the tallest student in a classroom. You cannot filter the students until you first measure everyone and calculate the average height. A subquery is exactly that: a query that runs first to find a dynamic threshold (like average salary), which is then passed to the outer query to filter the rows.

Suppose you want to find employees who earn more than the average salary of the entire company. Your first instinct might be to write this query:

sql — wrong_where_aggregate.sql
-- ❌ THIS WILL FAIL WITH A SYNTAX ERROR!
SELECT name, salary
FROM employees
WHERE salary > AVG(salary);  -- Aggregates not allowed in WHERE
⚠️ Why did it fail? (SQL Execution Order): The database engine runs the WHERE clause before it aggregates any data. Since the engine doesn't know what the overall AVG(salary) is when checking individual rows, it throws a syntax error.

To solve this, we nest a query inside another query. The inner query calculates the average first, and its single output value is handed to the outer query for row filtering:

sql — independent_subquery.sql
-- ✅ Independent Subquery: Runs exactly once first, then filters the outer query
SELECT name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary) FROM employees
);

-- Subquery with IN (finding employees who work in departments located in Mumbai)
SELECT name 
FROM employees
WHERE dept_id IN (
    SELECT id FROM departments WHERE location = 'Mumbai'
);

Correlated Subqueries — The Performance Bottleneck

Unlike independent subqueries which run once, a Correlated Subquery references columns from the outer query. The database engine must execute the inner query once for every single row processed by the outer query. This is physically slow and should be avoided on large datasets unless optimized.

sql — correlated_subquery.sql
-- Find employees earning more than the average salary of THEIR specific department
SELECT e1.name, e1.salary, e1.dept_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.dept_id = e1.dept_id  -- Binds inner query to the current outer row
);

2. CTEs (WITH clause) — Solving the "Spaghetti Query"

CTEs are like defining intermediate variables in Python (e.g. high_earning_df = df[df.salary > 70000]) before running a merge. Instead of nesting queries inside queries like Russian nesting dolls, you declare readable temporary blocks at the top of your script.

If you need to perform multiple levels of aggregations or joins on subqueries, nesting them quickly turns your SQL script into unreadable spaghetti code. For example:

sql — nested_spaghetti.sql
-- ❌ Nested Subqueries: Hard to read, scan, and maintain
SELECT d.department_name, ds.headcount
FROM (
    SELECT dept_id, COUNT(*) AS headcount
    FROM (
        SELECT name, salary, dept_id 
        FROM employees 
        WHERE salary > 70000
    ) AS high_earners
    GROUP BY dept_id
) ds
JOIN departments d ON ds.dept_id = d.id;

A Common Table Expression (CTE) solves this. By using the WITH clause, you declare named, temporary virtual tables at the start of your script and query them sequentially:

sql — cte_refactored.sql
-- ✅ Clean, modular, and self-documenting
WITH high_earners AS (
    SELECT name, salary, dept_id
    FROM employees
    WHERE salary > 70000
),
dept_summary AS (
    SELECT dept_id, COUNT(*) AS headcount
    FROM high_earners
    GROUP BY dept_id
)
SELECT d.department_name, ds.headcount
FROM dept_summary ds
JOIN departments d ON ds.dept_id = d.id
ORDER BY ds.headcount DESC;

2.1. Recursive CTEs — Tree Traversal

Standard SQL joins cannot query hierarchical structures (like company org charts or nested website categories) where the depth is unknown. Recursive CTEs loop through rows until no more connections are found.

📐 Deconstructing the Recursive CTE Pipeline: A recursive CTE contains three core parts connected via UNION ALL:
  1. Anchor Member: The base query that starts the recursion. It runs once and finds the root node (e.g. CEO who has no manager).
  2. Recursive Member: The query that joins the CTE back to the base table to fetch the next level (e.g. subordinates who report to the managers found in the previous iteration). It increments a tracker (like level).
  3. Termination Condition: This is implicit. The recursion stops automatically when an iteration returns 0 rows (no more subordinates left to join).
sql — recursive_org_chart.sql
WITH RECURSIVE org_chart AS (
    -- 1. Anchor Member: Find the top-level bosses (manager_id is NULL)
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL
    
    UNION ALL
    
    -- 2. Recursive Member: Join the CTE back to find direct reports
    SELECT e.id, e.name, e.manager_id, o.level + 1
    FROM employees e
    INNER JOIN org_chart o ON e.manager_id = o.id
)
SELECT * FROM org_chart ORDER BY level, name;

To use window functions, we use the OVER clause. It defines the "window" or partition of data the function calculates across. Let's break down the syntax step-by-step:

📐 Anatomy of a Window Definition:
FUNCTION() OVER (PARTITION BY department ORDER BY salary DESC)
• PARTITION BY: Slices the dataset into subgroups (like GROUP BY, but rows aren't collapsed).
• ORDER BY: Sorts the rows inside each partition so the function runs in a specific sequence (critical for rankings or running totals).

Ranking Functions: ROW_NUMBER vs. RANK vs. DENSE_RANK

Suppose three employees in the IT department earn the exact same salary ($80,000). How do different ranking functions rank them?

  • ROW_NUMBER(): Assigns a unique consecutive number to every row (1, 2, 3) ignoring ties.
  • RANK(): Assigns the same rank to ties, but skips the next rank numbers (1, 1, 1, 4).
  • DENSE_RANK(): Assigns the same rank to ties without skipping numbers (1, 1, 1, 2).
sql — window_rankings.sql
SELECT name, department, salary,
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
    DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dense_rnk
FROM employees;

Time-Series Row Comparisons: LEAD and LAG

LEAD() fetches a value from a subsequent row, and LAG() fetches a value from a preceding row. They allow you to compare rows sequentially without expensive self-joins (perfect for Month-over-Month changes).

sql — lead_lag.sql
SELECT month, revenue,
    LAG(revenue, 1) OVER (ORDER BY month) AS prev_month_revenue,
    revenue - LAG(revenue, 1) OVER (ORDER BY month) AS mom_change
FROM monthly_sales;

4. CASE WHEN & COALESCE — Row-Level Logic

SQL provides standard inline conditional statements to alter data values on a row-by-row basis:

sql — case_coalesce.sql
-- 1. CASE WHEN: The IF-ELSE statement of SQL
SELECT name, salary,
    CASE
        WHEN salary > 80000 THEN 'Senior'
        WHEN salary > 50000 THEN 'Mid-level'

5. EXISTS vs. IN — Performance & Null Behaviors

To check if a value exists in another set, we use IN or EXISTS. But they act differently under the hood:

  • IN: The database retrieves the entire subquery result set, caches it in memory, and checks if the outer value exists in that set.
  • EXISTS: The database uses a semi-join. It stops searching the moment it finds the first matching record in the subquery. For large subqueries, EXISTS is significantly faster!
⚠️ The NOT IN Null Trap: If the subquery returns even a single NULL, a query using NOT IN will return zero rows. This is because SQL evaluates x NOT IN (y, NULL) as UNKNOWN. To prevent this, always use NOT EXISTS, which handles NULLs safely and correctly!
sql — exists_vs_in.sql
-- Use IN for small, static lists (e.g. IN (1, 2, 3))
SELECT name FROM employees
WHERE dept_id IN (
    SELECT id FROM departments WHERE location = 'Mumbai'
);

-- Use EXISTS for large subqueries (Faster execution)
SELECT e.name FROM employees e
WHERE EXISTS (
    SELECT 1 FROM departments d
    WHERE d.id = e.dept_id AND d.location = 'Mumbai'
);

6. Running Aggregates — Dynamic Windows

By ordering inside the OVER clause, we can compute cumulative and rolling metrics over time:

sql — running_aggregates.sql
SELECT month, revenue,
    SUM(revenue) OVER (ORDER BY month) AS running_total,
    AVG(revenue) OVER (
        ORDER BY month
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS rolling_3mo_avg,
    NTILE(4) OVER (ORDER BY revenue) AS revenue_quartile,
    PERCENT_RANK() OVER (ORDER BY revenue) AS percentile_rank
FROM monthly_sales;

7. Views & Materialized Views — Virtual vs. Cached Queries

A standard View is like a "saved filter" on your inbox. No emails are copied; it just reruns the search on-the-fly when you click it. A Materialized View is like taking a screenshot of your inbox. It is extremely fast to look at, but it doesn't show new emails until you manually take a new screenshot (Refresh).

To reuse code and restrict access to sensitive table columns, databases let you save queries as views:

PerformanceSlower (Must run query from scratch each time)Extremely Fast (Reads precomputed rows directly)
Data FreshnessAlways Fresh (Retrieves real-time table state)Stale until updated manually/automatically

-- 1. Create a Standard View
CREATE VIEW v_high_earners AS
    SELECT name, department, salary
    FROM employees
    WHERE salary > 70000;

-- 2. Create a Materialized View (Requires REFRESH to update)
CREATE MATERIALIZED VIEW mv_sales_summary AS
    SELECT order_date, COUNT(*) AS total_orders, SUM(amount) AS daily_revenue
    FROM orders
    GROUP BY order_date;

-- Refreshing the cached data in the materialized view
REFRESH MATERIALIZED VIEW mv_sales_summary;

8. Database Triggers — Event-Driven Automation

Triggers are like smart home automations: "IF motion sensor detects movement BEFORE 6 AM, THEN turn on the hallway lights". In databases: "IF a row is updated in table X, THEN log the historical values to audit table Y".

A Trigger automatically executes a procedure in response to INSERT, UPDATE, or DELETE events on a specific table. This is perfect for security auditing and maintaining history logs:

sql — triggers.sql
-- 1. Audit log table
CREATE TABLE salary_audit_log (
    emp_id INT,
    old_salary INT,
    new_salary INT,
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. Trigger Function (PostgreSQL PL/pgSQL syntax)
CREATE OR REPLACE FUNCTION log_salary_change()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.salary <> OLD.salary THEN
        INSERT INTO salary_audit_log(emp_id, old_salary, new_salary)
        VALUES(OLD.id, OLD.salary, NEW.salary);
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 3. Bind the trigger to run BEFORE the update executes on each row
CREATE TRIGGER trg_salary_update
BEFORE UPDATE ON employees
FOR EACH ROW
EXECUTE FUNCTION log_salary_change();

Cursors — Row-by-Row Iteration

While standard SQL works on sets of rows simultaneously, a Cursor is a pointer that allows you to iterate through query results row-by-row, running loops and conditional operations in a procedural container.

sql — cursors.sql
CREATE OR REPLACE FUNCTION distribute_bonuses() RETURNS VOID AS $$
DECLARE
    emp_record RECORD;
    emp_cursor CURSOR FOR SELECT id, salary FROM employees WHERE perf_score >= 9;
BEGIN
    OPEN emp_cursor;
    LOOP
        FETCH NEXT FROM emp_cursor INTO emp_record;
        EXIT WHEN NOT FOUND;
        
        UPDATE employees 
        SET salary = salary + (emp_record.salary * 0.10) 
        WHERE id = emp_record.id;
    END LOOP;
    CLOSE emp_cursor;
END;
$$ LANGUAGE plpgsql;
⚠️ The Cursor Performance Trap: Writing cursor loops is slow because it bypasses the database compiler's query optimization. A single set-based query like UPDATE employees SET salary = salary * 1.10 WHERE perf_score >= 9; runs up to 100x faster than a cursor. Use cursors only when row-by-row procedural calculation is absolutely unavoidable.

10. Transactions & ACID Properties

Transactions bundle multiple SQL writes into a single, atomic block. Either all operations succeed, or none do. This preserves database integrity in the event of hardware failures or network cuts.

  • Atomicity: All statements in a transaction succeed, or everything is rolled back (All-or-Nothing).
  • Consistency: The database transitions cleanly according to rules (e.g. constraints, unique keys).
  • Isolation: Concurrent transactions run independently and do not read uncommitted states.
  • Durability: Once committed, updates are permanently written to disk and survive system crashes.
sql — transactions.sql
BEGIN;
    UPDATE accounts SET balance = balance - 1000 WHERE id = 1;  -- Debit
    UPDATE accounts SET balance = balance + 1000 WHERE id = 2;  -- Credit
COMMIT;   -- Persists changes on disk

-- If an operation fails, run ROLLBACK to revert to initial state:
BEGIN;
    DELETE FROM orders WHERE status = 'cancelled';
ROLLBACK;  -- Reverts the deletes

11. Database Locks — Shared vs. Exclusive

A Shared Lock is like sharing a view-only document: many people can read it simultaneously, but no one can edit it. An Exclusive Lock is like opening a document in edit-only checkout mode: one person gets full write access, and everyone else is blocked from editing or reading until the document is closed.
sql — locks.sql
-- 1. Shared Lock (FOR SHARE): Allows others to read, blocks write updates
SELECT balance FROM accounts WHERE id = 1 FOR SHARE;

-- 2. Exclusive Lock (FOR UPDATE): Blocks others from both reading and writing
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;

15. pgvector — Modern Vector Search & Similarity

Keyword search using operators like LIKE '%python%' has a major limitation: it only returns exact character matches. If a user searches for "machine learning", keyword queries fail to match documents containing "ML", "neural networks", or "deep learning" because the terms are textually different, despite being semantically similar.

Keyword search is like sorting clothes by looking at label text (e.g. "red shirts"). Vector search is like showing the salesperson a shirt and saying "find me fabrics that feel and look similar to this" (semantic similarity distance).

Modern LLM applications (like Retrieval-Augmented Generation or RAG) convert text blocks into high-dimensional numerical lists called vector embeddings. The pgvector extension allows PostgreSQL to store, index, and query vectors directly alongside your relational tables:

In RAG architectures, we embed a user's natural language question, run a vector similarity query in PostgreSQL to retrieve the most relevant background documents, and pass that retrieved text as context to LLMs like Claude or GPT.
sql — pgvector_search.sql
-- 1. Enable pgvector extension
CREATE EXTENSION IF NOT EXISTS vector;

-- 2. Create table storing text alongside 1536-dimensional embeddings (OpenAI standard)
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding VECTOR(1536)
);

-- 3. Perform Cosine Similarity vector search to find top 3 matching documents
SELECT content, 1 - (embedding <=> '[0.012, -0.004, ..., 0.89]') AS cosine_similarity
FROM documents
ORDER BY embedding <=> '[0.012, -0.004, ..., 0.89]' ASC
LIMIT 3;

-- Distance Operators:
-- <=> Cosine Distance (used for textual semantic similarity)
-- <-> L2 Euclidean Distance (used for physical distance/coordinate metrics)
-- <#> Negative Inner Product
⚡ Vector Indexing (HNSW vs IVFFlat): By default, similarity queries perform a full table scan. To speed up vector queries:
• IVFFlat: Divides vectors into clusters. Fast index build times, but slightly lower recall.
• HNSW (Hierarchical Navigable Small World): Creates a multi-layer graph. Extremely fast queries and high recall, but takes longer to build and consumes more memory.

⚡ Key Takeaways — Day 12

  1. Window functions compute across rows related to the current row without collapsing them. ROW_NUMBER(), RANK(), DENSE_RANK() are essential for interview problems.
  2. CTEs (WITH clause) make complex queries readable. Recursive CTEs loop back on themselves using an Anchor Member and a Recursive Member, making them ideal for hierarchical or tree traversal queries.
  3. Views are virtual tables executing queries dynamically. Materialized Views cache physical data for faster queries but require manual updates using REFRESH MATERIALIZED VIEW.
  4. Triggers are event-driven database procedures executing automatically BEFORE or AFTER database write events (INSERT, UPDATE, DELETE).
  5. Stored Procedures vs. Functions: Stored Procedures (`CALL`) are used for business workflows and support transaction control (`COMMIT`/`ROLLBACK`). User-Defined Functions (`SELECT`) must return a value and cannot control transactions.
  6. Cursors iterate result sets row-by-row and are generally slower than optimized **set-based SQL** operations, which the database engine executes in parallel.
  7. pgvector Similarity Search: Relational databases can perform vector search natively using `pgvector`. This uses distance operators like Cosine Distance (<=>) or L2 Distance (<->) and indexes (IVFFlat/HNSW) to retrieve semantically matching text embeddings (crucial for AI/ML RAG architectures).
  8. Concurrency & Locks: Relational databases manage concurrent access using Shared (FOR SHARE) or Exclusive (FOR UPDATE) locks and isolation levels (Read Committed to Serializable) to enforce ACID consistency.

🛠️ WEEKLY DEBUGGING CHALLENGE

Debug: Broken Aggregate Query

The query fails when trying to count orders placed by each customer.

python
-- Select customer ID and count of their orders
SELECT customer_id, COUNT(order_id)
FROM orders
WHERE order_date >= '2026-01-01';
SOLUTION & EXPLANATION
The Bug: When using aggregate functions (like COUNT, SUM, AVG) alongside a non-aggregate column (like customer_id), you must group by the non-aggregate column. Otherwise, the database engine does not know how to associate the aggregate calculation with individual records.

The Fix: 
SELECT customer_id, COUNT(order_id)
FROM orders
WHERE order_date >= '2026-01-01'
GROUP BY customer_id;
